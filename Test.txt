protocole expérimental version 1993 : 1459 (https://www.rfc-editor.org/rfc/rfc1459.html)

protocole informationnnel version 2000 : 2812 (https://www.rfc-editor.org/rfc/rfc2812)

FORMAT DES MESSAGES :

	:<prefix><space><command><space><params>

avec prefix :

	<servername><nick>!<user>@host

<channel> inclut #

ATTENTION :

	- HEXCHAT accepte / x espace COMMANDE

PASS tests :

Un mdp aura préalablement été défini au lancement du serveur.

	- syntaxe : PASS <mdp>

✅	- vérifier la bonne prise en compte des "mdp" et 'mdp' et "'mdp'" et '"mdp"'

✅	- un client s'identifie sans mdp : PASS
		-> un message s'affiche dans le terminal du client (ERR_NEEDMOREPARAMS) :
		": 461 !@localhost PASS :Not enough parameters"
		NB : pas de différence entre les versions
		-> message ft_IRC :
		": 461 !@localhost PASS :Not enough parameters"

✅	- un client s'identifie avec un mdp constitué de plusieurs espaces (les espaces sont skip)
		-> un message s'affiche dans le terminal du client (ERR_NEEDMOREPARAMS) :
		": 461 !@localhost PASS :Not enough parameters"
		NB : pas de différence entre les versions
		-> message ft_IRC :
		": 461 !@localhost PASS :Not enough parameters"

✅	- un client s'identifie avec un mdp faux :
		-> un message s'affiche dans le terminal du client (ERR_PASSWDMISMATCH) :
		": 464 !@localhost :Password incorrect"
		NB : pas de différence entre les versions
		-> message ft_IRC :
		": 464 !@localhost :Password incorrect"

✅	- un client s'identifie avec la commande PASS suivie de plusieurs espaces puis du mdp
	correct :
		-> les espaces sont skip et le mdp est accepté

✅	- un client s'identifie avec la commande PASS précédée d'un ou plusieurs espaces puis du
	mdp correct :
		-> aucun message ne s'affiche dans le terminal du client
		-> le client doit saisir son mdp
		-> HEXCHAT EFnet reconnaît la commande si les espaces sont placés après le "/"
		-> ft_IRC :
		pas de message

✅	- un client s'identifie avec la commande PASS suivie du mdp correct et de plusieurs autres
	mots :
		-> seul le premier mot (le mdp) est pris en compte

✅	- un client a déjà saisi le bon mdp :
		-> un message s'affiche dans le terminal du client (ERR_ALREADYREGISTRED) :
		~ version 2812 :
		": 462 !@localhost :Unauthorized command (already registered)"
		~ version 1459 :
		": 462 !@localhost :You may not reregister"
		-> ft_IRC :
⚠️		": 462 !@localhost :Unauthorized command (already registered)"

✅	- un client s'identifie en saisissant son nickname avant son mdp :
		-> aucun message ne s'affiche dans le terminal du client
		-> le client doit saisir son mdp

✅	- un client s'identifie en saisissant son username avant son mdp ou avant son nickname :
		-> aucun message ne s'affiche dans le terminal du client
		-> le client doit saisir son mdp ou son nickname

NICK tests :

	- syntaxe : NICK <nickname>
	
	- caractères spéciaux : "[", "]", "\", "`", "_", "^", "{", "|", "}"

		": 431 !@localhost * :No nickname given"
		NB : pas de différence entre les versions
		-> message ft_IRC :
		": 431 !@localhost * :No nickname given"

	- un client s'identifie avec un nickname qui ne répond pas aux règles. Ces règles sont les
	suivantes :
		~ version 2812 :
✅			- 1 caractère alphabétique ou spécial au début,
			- au maximum 8 caractères alphabétiques, spéciaux ou numériques ensuite,
		~ version 1459 :
			- 1 caractère alphabétique ou spécial au début,
			- des caractères alphabétiques, spéciaux ou numériques ensuite,
		
		-> un message s'affiche dans le terminal du client (ERR_ERRONEUSNICKNAME) :
✅		": 432 !@localhost <nick> :Erroneus nickname"
		NB : pas de différence entre les versions en ce qui concerne les messages

		~ version 2812 :
✅			- les caractères au delà du 9ème ne sont pas pris en compte, et aucune
			erreur ne s'affiche.
		~ version 1459 :
			- pas de restriction

✅	- un client s'identifie avec la commande NICK suivie de plusieurs espaces puis du
	nickname :
		-> les espaces sont skip
	
✅	- un client s'identifie avec la commande NICK précédée d'un ou plusieurs espaces puis du
	nickname :

		-> le client doit saisir son nickname

✅	- un client s'identifie en saisissant son nickname avant son mdp :

		-> le client doit saisir son mdp

✅	- un client s'identifie avec un nickname déjà utilisé :

✅	- un client quitte le serveur et se reconnecte avec son nickname précédent :
		-> c'est possible

PRIVMSG tests :

	- syntaxe : PRIVMSG <target> <message>
	avec <message> pouvant contenir des espaces si le premier caractère est ':'
	avec <target> est un autre user ou un channel
	
✅	- un user envoie un message sans arguments : PRIVMSG
		-> un message s'affiche dans le terminal du client (ERR_NORECIPIENT) :
		": 411 <nick>!<user>@localhost :No recipient given (PRIVMSG)"
		NB : pas de différence entre les versions
		NB : cette erreur devrait s'appliquer quand le destinataire n'a pas été spéficié mais
		il semblerait que HEXCHAT EFnet considère que le message est absent avec la commande
		suivante : PRIVMSG :message
		NB : pas de différence entre les versions
		-> message ft_IRC :
		": 411 <nick>!<user>@localhost :No recipient given (PRIVMSG)"

✅	- un user envoie un message sans spécifier de destinataire : PRIVMSG :<message>
		-> un message devrait s'afficher dans le terminal du client (ERR_NORECIPIENT) :
		": 411 <nick>!<user>@localhost :No recipient given (PRIVMSG)"
		NB : cette erreur devrait s'appliquer quand le destinataire n'a pas été spéficié mais
		il semblerait que HEXCHAT EFnet considère que le message est absent avec la commande
		suivante : PRIVMSG :message
⚠️		-> pas de message à envoyer ": 412 <nick>!<user>@localhost :No text to send"
